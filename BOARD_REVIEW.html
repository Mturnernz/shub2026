Executive Summary

What we found

A technically sound platform with a coherent design system, correct legal/safety architecture, and well-structured database — undermined by execution gaps in conversion, trust signalling, and brand voice. None of the issues require a redesign. All require focused completion.

Where we stand

Phases 1–5 are complete. Eight UX flow gaps identified in the audit have been shipped. Five P1 actions remain — the most critical being provider accept/decline (the core marketplace loop) and the post-booking confirmation screen.

Already Shipped — 8 UX Flow Fixes

What's been fixed

01
Password reset completed
New /update-password screen handles the Supabase recovery token. Previously the reset flow was a dead end after the email was sent.
02
Account guest state made reachable
Removed ProtectedRoute from /account. Guest "Join shub" UI (Create account / Sign in) is now visible to unauthenticated users.
03
Browse-as-guest path added
"Browse without signing up" link added to the landing screen. Guests can now reach /discover without going through auth.
04
Login-first friction removed
"I'm looking" now routes to /signup/client, not /login. New users see signup first. Existing users use the "Already have an account?" link.
05
"Both" role added to landing
Third role card "I'm both" added. Routes through client signup then provider onboarding. Role merging fixed in ProviderOnboarding to preserve existing client role.
06
Email confirmation screen
New /welcome screen with "Start browsing" CTA. Previously users landed on /discover silently after confirming their email.
07
Three broken Account links fixed
My arrangements → /bookings (new screen) · Saved companions → /saved (new screen) · Privacy settings → /privacy (new screen). All three previously broken or self-linking.
08
"Become a companion" path added
Client-only users now see a "Become a companion" CTA in Account. Links to provider onboarding. Role merge logic preserves their existing client access.

What's next

Remaining P1 actions

These five items are the next build sprint before public launch. Each is scoped, with the affected file and estimated complexity noted.

🔴
P1 — Critical
Booking confirmation screen
After payment/request, show a confirmation screen instead of a toast. Include provider chip, booking summary, "What happens next" timeline, and Message CTA.
src/flows/Book.tsx
🔴
P1 — Critical
Provider accept / decline UI
Dashboard pending cards need Accept and Decline action buttons wired to a Supabase booking status update. Without this, the marketplace loop never closes.
src/screens/provider/Dashboard.tsx
🟡
P1 — High
Review submission flow
Add "Leave a review" button on completed booking cards. Opens a bottom sheet with 5-star tap-to-rate and optional text. DB trigger already updates provider rating on insert.
src/screens/client/Bookings.tsx
🟡
P1 — High
Initials avatar system
Replace emoji placeholder in ProviderCard with a two-letter monogram rendered on a gradient derived from the listing's bg_from/bg_to colours. Eliminates the "toy app" perception immediately.
src/components/cards/ProviderCard.tsx
🟡
P1 — High
Cold-start / empty Discover
When provider count is below threshold, replace empty grid with an email-capture waitlist state: "We're launching soon in [city]. Be first to know." Prevents first-visit bounce.
src/screens/client/Discover.tsx

Booking loop close rate Pending bookings that reach confirmed status — > 70% Provider Accept/Decline UI on Dashboard
Post-booking message rate Clients who message provider after confirmation — > 80% Booking confirmation screen with Message CTA
